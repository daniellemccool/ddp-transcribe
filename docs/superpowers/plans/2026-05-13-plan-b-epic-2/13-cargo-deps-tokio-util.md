# Task 13 — Add `tokio-util = { version = "0.7", features = ["sync"] }` to Cargo.toml

**Goal:** Add the `tokio-util` direct dependency Phase 2 needs for `CancellationToken`. The spec's claim that tokio-util is "transitively present" is wrong (verified at plan-write time — `grep 'tokio-util' Cargo.lock` empty); this task adds it explicitly. Matches Epic 1's convention of a clean standalone cargo-deps task (Epic 1 T2 = cargo deps for whisper-rs + hound; this task = cargo deps for tokio-util).

- [ ] **Step 1: Confirm `tokio-util` is not present**

```bash
grep -n 'tokio-util\|tokio_util' Cargo.toml Cargo.lock
```

Expected: nothing in Cargo.toml; nothing in Cargo.lock. If `tokio-util` IS already present (a previous task added it), skip Steps 2–4 and just verify the `sync` feature is enabled.

- [ ] **Step 2: Add to `Cargo.toml`**

In the `[dependencies]` section, append:

```toml
tokio-util = { version = "0.7", features = ["sync"] }
```

Caret-pin (`"0.7"`) is intentional: tokio-util's `0.7` line tracks tokio's `1.x` line; the `sync::CancellationToken` API is stable on this line. Patch upgrades within `0.7.*` flow automatically.

The `sync` feature gates `CancellationToken`. No other tokio-util features are needed (`codec`, `io`, `compat` stay off).

> **Note (post-implementation correction, ADR 0003 deviation honesty):** tokio-util 0.7 has no `sync` feature gate — the `sync` module (`CancellationToken` included) is compiled unconditionally. The T13 implementer discovered this at commit time: `cargo add tokio-util --features sync` silently succeeds but the feature key is a no-op. The committed `Cargo.toml` line is `tokio-util = "0.7"` (no features array) per the implementer's deviation disclosure. The feature-array form above is an artifact of plan-write-time API assumption; it should not be used as a template for future Cargo edits. Verify against the installed crate's `Cargo.toml` first (per the cross-epic library-API drift FOLLOWUP).
